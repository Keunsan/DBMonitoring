# 통합 DB 모니터링 시스템 요구사항 정의서

## 1. 문서 개요

### 1.1 목적

본 문서는 MSSQL, Oracle, Azure SQL Database 등 사내 주요 데이터베이스를 통합 모니터링하기 위한 시스템의 기능 요구사항, 비기능 요구사항, 데이터 요구사항, 권한 요구사항을 정의한다.

### 1.2 적용 범위

- On-premise MSSQL
- On-premise Oracle
- Azure SQL Database
- 향후 PostgreSQL, MySQL, MariaDB 등 확장 가능한 구조 고려

### 1.3 주요 사용자

- DBA
- 시스템 운영자
- 개발 담당자
- 보안 담당자
- 시스템 관리자
- IT 관리 책임자

## 2. 업무 요구사항

### BR-001. 이기종 DB 통합 모니터링

시스템은 MSSQL, Oracle, Azure SQL Database의 상태를 하나의 포털에서 통합적으로 조회할 수 있어야 한다.

수용 기준:

- DBMS 유형별로 수집 항목을 구분할 수 있어야 한다.
- 업무 시스템, DB 그룹, 중요도, 담당자 기준으로 조회할 수 있어야 한다.
- 전체 DB 상태를 정상, 주의, 경고, 장애 등급으로 표시해야 한다.

### BR-002. 실시간 장애 징후 탐지

시스템은 성능 저하, 락, 교착상태, 장시간 실행 쿼리, 세션 급증 등 장애 징후를 실시간으로 탐지해야 한다.

수용 기준:

- 주요 성능 지표를 최소 5초에서 60초 주기로 수집할 수 있어야 한다.
- 임계치 초과 시 이벤트를 생성해야 한다.
- 동일 이벤트 반복 발생 시 알림 중복 억제 정책을 적용할 수 있어야 한다.

### BR-003. SQL 성능 이력 분석

시스템은 과거 실행 SQL의 성능 이력을 저장하고, 성능 저하 SQL을 자동으로 식별해야 한다.

수용 기준:

- SQL별 실행 횟수, 평균 수행 시간, 최대 수행 시간, CPU 사용량, I/O 사용량을 저장해야 한다.
- 기준 성능 대비 악화된 SQL을 탐지해야 한다.
- 실행 계획 변경 전후 성능을 비교할 수 있어야 한다.

### BR-004. 보안 및 계정 관리 모니터링

시스템은 DB 계정, 권한, 접속 실패, 권한 변경 이력을 모니터링해야 한다.

수용 기준:

- 장기 미사용 계정을 탐지할 수 있어야 한다.
- 고위험 권한 보유 계정을 식별할 수 있어야 한다.
- 권한 변경 이력과 로그인 실패 이력을 조회할 수 있어야 한다.

### BR-005. 알림 및 이슈 관리

시스템은 성능 또는 보안 이벤트 발생 시 담당자에게 알림을 발송하고, 이슈 처리 상태를 관리해야 한다.

수용 기준:

- DB별, 업무별, 이벤트 유형별 담당자를 지정할 수 있어야 한다.
- 알림 채널을 Email, SMS, Teams, Slack, 사내 메신저 등으로 확장할 수 있어야 한다.
- 이슈 상태를 신규, 확인, 조치중, 완료, 보류로 관리할 수 있어야 한다.

## 3. 기능 요구사항

### FR-001. 통합 대시보드

설명:

전체 DB 운영 상태를 한 화면에서 요약 제공한다.

상세 요구사항:

- 전체 DB 수, 정상 DB 수, 경고 DB 수, 장애 DB 수 표시
- DBMS별 상태 분포 표시
- 업무 시스템별 DB 상태 표시
- 최근 알림 및 미처리 이슈 표시
- 리소스 사용률 상위 DB 표시
- 장시간 실행 쿼리 및 Blocking 발생 DB 표시

우선순위: 필수

### FR-002. DB 인스턴스 관리

설명:

모니터링 대상 DB를 등록하고 관리한다.

상세 요구사항:

- DBMS 유형, 호스트, 포트, 서비스명, 데이터베이스명 등록
- 업무 시스템, 중요도, 운영 구분, 담당자 매핑
- 수집 계정 또는 인증 정보 등록
- 수집 활성화 및 비활성화 관리
- 수집 주기 설정
- 연결 테스트 기능 제공

우선순위: 필수

### FR-003. 실시간 성능 모니터링

설명:

DB별 성능 지표를 실시간으로 조회한다.

상세 요구사항:

- CPU 사용률
- 메모리 사용률
- 디스크 I/O
- 네트워크 I/O
- 세션 수
- Active Session 수
- 대기 세션 수
- 트랜잭션 수
- 초당 요청 수
- 로그 사용률
- Temp 영역 사용률
- DB별 특화 지표 표시

우선순위: 필수

### FR-004. 실시간 세션 모니터링

설명:

현재 DB에 접속 중인 세션과 실행 중인 SQL을 조회한다.

상세 요구사항:

- 세션 ID, 사용자, 프로그램, 호스트, 상태 표시
- 현재 실행 SQL 표시
- 실행 시간, CPU, I/O, Wait 정보 표시
- Blocking 세션과 Blocked 세션 관계 표시
- 장시간 실행 세션 강조
- 필요 시 세션 종료 요청 기능은 권한 기반으로 제공

우선순위: 필수

### FR-005. 락 및 교착상태 모니터링

설명:

락 경합과 교착상태를 탐지하고 원인을 분석한다.

상세 요구사항:

- Blocking Tree 표시
- 락 보유 세션과 대기 세션 표시
- 락 대상 객체 표시
- Deadlock 발생 이력 저장
- Deadlock 관련 SQL과 세션 정보 연결
- 반복 발생 패턴 분석

우선순위: 필수

### FR-006. Wait 분석

설명:

DB 성능 저하의 원인을 Wait 기반으로 분석한다.

상세 요구사항:

- MSSQL Wait Type 수집 및 분석
- Oracle Wait Event 수집 및 분석
- 시간대별 Wait 추이 표시
- DB별 Top Wait 표시
- SQL, 세션, 리소스 지표와 Wait 연계

우선순위: 필수

### FR-007. SQL 성능 이력 관리

설명:

실행 SQL의 성능 데이터를 수집하고 장기 보관한다.

상세 요구사항:

- SQL ID 또는 Query Hash 기준 집계
- 실행 횟수, 평균 수행 시간, 최대 수행 시간 수집
- CPU 시간, Logical Read, Physical Read, 쓰기량 수집
- 실행 계획 식별자 저장
- SQL Text 저장 시 마스킹 정책 적용
- 기간별 Top SQL 조회

우선순위: 필수

### FR-008. 실행 계획 변경 분석

설명:

SQL 실행 계획 변경 시점과 성능 영향을 분석한다.

상세 요구사항:

- SQL별 실행 계획 이력 저장
- 변경 전후 수행 시간, CPU, I/O 비교
- 성능 악화 Plan 식별
- 인덱스 변경, 통계 변경, 파라미터 영향 후보 표시
- Plan 고정 또는 튜닝 검토 권고

우선순위: 중요

### FR-009. 성능 회귀 탐지

설명:

정상 성능 기준 대비 악화된 SQL과 DB 지표를 자동 탐지한다.

상세 요구사항:

- SQL별 Baseline 자동 산정
- 최근 성능과 기준 성능 비교
- 평균 수행 시간, CPU, I/O, 실행 횟수 급증 탐지
- 반복 발생 여부 표시
- 이슈 자동 생성 연계

우선순위: 중요

### FR-010. 개선 권고

설명:

성능 이슈에 대해 원인 후보와 조치 방향을 제시한다.

상세 요구사항:

- 인덱스 누락 후보 제시
- 통계 갱신 필요 후보 제시
- Full Scan, Sort, Hash Join, Spill 의심 항목 표시
- 파라미터 스니핑 의심 SQL 표시
- 쿼리 재작성 검토 포인트 제시
- 권고 결과에 대한 조치 이력 관리

우선순위: 중요

### FR-011. 계정 및 권한 모니터링

설명:

DB 계정과 권한 상태를 점검한다.

상세 요구사항:

## 4. 비기능 요구사항

### NFR-001. 성능

- 실시간 대시보드는 기본 조회 기준 3초 이내 응답해야 한다.
- 수집 데이터 적재는 수집 주기 내 처리되어야 한다.
- 대량 SQL 이력 조회 시 기간, DB, SQL 조건 기반 페이징을 제공해야 한다.

### NFR-002. 확장성

- 신규 DBMS 추가가 가능한 수집 모듈 구조를 제공해야 한다.
- DB 인스턴스 증가에 따라 수집 서버를 수평 확장할 수 있어야 한다.
- 알림 채널 추가가 가능한 연동 구조를 제공해야 한다.

### NFR-003. 안정성

- 수집 실패 시 재시도 및 실패 이력 저장이 가능해야 한다.
- 중앙 저장소 장애 시 수집 데이터 유실을 최소화하는 버퍼링 방식을 검토해야 한다.
- 알림 발송 실패 시 재전송 또는 실패 이력 확인이 가능해야 한다.

### NFR-004. 보안

- DB 접속 계정 정보는 암호화 저장해야 한다.
- 사용자 인증은 사내 SSO 또는 AD 연동을 검토한다.
- 권한 기반 접근 제어를 적용해야 한다.
- SQL Text 내 개인정보 또는 민감정보 마스킹 정책을 제공해야 한다.
- 시스템 접속 및 주요 설정 변경은 감사 로그로 남겨야 한다.
- 모든 개발은 시큐어 코딩 원칙을 기본으로 적용해야 한다.
- 사용자 입력, API 요청, DB 연결 설정, 수집 대상 정보, 필터 조건은 서버 측에서 검증하고 허용 목록 기반으로 처리해야 한다.
- SQL 실행, DMV 조회, 리포트 조회 등 DB 접근 로직은 파라미터 바인딩 또는 안전한 쿼리 빌더를 사용해야 하며 문자열 결합 방식의 동적 SQL 생성을 금지한다.
- 인증, 세션, 토큰, 쿠키는 안전한 만료 정책, 재사용 방지, Secure/HttpOnly/SameSite 설정을 적용해야 한다.
- 비밀번호, API Key, DB 접속 문자열, 인증 토큰 등 비밀 정보는 소스 코드와 로그에 남기지 않고 안전한 비밀 저장소 또는 환경 변수로 관리해야 한다.
- 화면과 API 응답은 권한 범위에 맞는 데이터만 반환해야 하며, 클라이언트 숨김 처리에 의존하지 않아야 한다.
- 관리자 기능, 세션 종료, 권한 변경, 알림 정책 변경, DB 등록/수정 등 고위험 기능은 세분화된 권한 확인과 감사 로그를 적용해야 한다.
- 오류 메시지는 내부 경로, 쿼리, 접속 정보, 스택 트레이스 등 민감 정보를 노출하지 않아야 한다.
- 의존성은 알려진 취약점 점검을 수행하고, 보안 패치 적용이 가능한 구조로 관리해야 한다.
- 배포 전 정적 분석, 의존성 취약점 점검, 주요 인증/권한/입력 검증 테스트를 수행해야 한다.

### NFR-005. 운영성

- 수집 상태, 수집 지연, 수집 오류를 모니터링할 수 있어야 한다.
- 임계치, 수집 주기, 보관 기간은 관리 화면에서 변경 가능해야 한다.
- 시스템 장애 발생 시 원인 확인을 위한 로그를 제공해야 한다.

### NFR-006. 호환성

- 웹 브라우저는 Chrome 최신 버전을 우선 지원한다.
- 사용자 화면은 1920x1080 이상 해상도에 최적화한다.
